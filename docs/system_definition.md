# ポケモンさんすう V2 システム定義書
*最終更新日時: 2026年6月1日 17:30 JST (マスターデータ構築検証・図鑑エラー検知フェーズ)*

## 1. プロジェクト目的
* 算数クイズを解くことで1〜9世代のポケモンを順次ゲットし、全世代コンプリートを目指す「世代進行型学習アプリ」へ刷新する。
* 学習進行度に応じてポケモンを段階的に解放し、長期的な学習意欲を向上させる。

## 2. 現状システム (V1.5) の機能要件
* **ユーザー管理**: 事前定義されたユーザー（りょうま、さら、ゲスト）を選択してプレイする形式。
* **難易度設定**: レベル1（1桁の足し引き）からレベル4（九九）までの4段階のクイズレベルが存在する。
* **クイズシステム**: 1回につき5問出題され、正解を含む4択から解答を選ぶ。
* **報酬ロジック**: 4問正解で1匹、5問全問正解で2匹のポケモンを獲得できる。
* **出現制限**: 選択した難易度によって出現するポケモンの範囲が制限される（レベル1・2は100番まで、レベル3は143番まで、レベル4は151番まで）。
* **図鑑機能**: 初代151匹固定の図鑑を表示し、未取得のポケモンは「？」で隠される。取得済みポケモンはモーダルで詳細（高さ、重さ、タイプ、説明文）を閲覧できる。

## 3. 現状システム (V1.5) の構成と課題
* **データ取得の負荷**: 起動時にPokeAPIから初代151匹分のデータを直接クライアント側で全件フェッチしているため、通信負荷と待機時間が発生している。
* **拡張性のないデータ構造**: Firestoreの `users` コレクション内のドキュメントに、取得したポケモンのIDを `owned` という単一の配列としてフラットに保存している。
* **ハードコードされた判定**: コンプリート判定が「取得した一意のIDの数が151に達したか」という固定値で組まれており、世代ごとの動的な変化に対応できない。
* **次世代機能の未実装**: 151匹コンプリート時に「つぎの せだいへ！」ボタンが表示されるが、押下時の処理はログ出力のみで実体がない。

## 4. マスターデータ構築（移行準備）の目的とデータ構造
* **目的**: クライアント起動時のPokeAPI直接呼び出しを廃止し、事前にAPIから取得・成形したデータをFirestoreにキャッシュすることで、アプリの起動速度と安定性を向上させる。ユーザーは自身の `current_gen` に対応する世代データのみを読み込む。
* **データ構造**:
    * **ドキュメントパス**: `masters/gen_{1..9}`
    * **フィールド**: `pokemon_list` (Array)
    * **配列内のオブジェクト構造（構築・取得済）**:
        * `local_id` (Number): 世代内の連番ID（1, 2, 3...）
        * `api_id` (Number): PokeAPIの全国図鑑No (`id`)
        * `name` (String): ポケモンの名前（日本語）
        * `image` (String): 公式アートワーク等の画像URL
        * `height` (Number): 高さ
        * `weight` (Number): 重さ
        * `types` (Array): 日本語変換済みのタイプ一覧
        * `flavor` (String): 日本語の説明文（フレーバーテキスト） **※注意：現在、漢字混じりのテキスト（`ja`）が優先取得・格納されているため、子供向けUI適応のために後段で「ひらがな最優先（`ja-Hrkt`）」への上書き再構築が必要。**

## 5. 要求仕様 (Requirements)
* **独立性の原則（完全不可侵）**: 
    * 既存の v1 環境（`index_v1.5` および Firestore `users` コレクション）に対し、削除・更新・変更等の干渉を一切行わない。
    * v1 データは移行時の参照（`get`）以外、一切のアクセスを禁止する。
* **システム分離**: V2 の運用データは `users_v2` コレクションで完全に独立管理する。
* **データ移行プロトコル**: 
    * ログイン時の自動データ同期は禁止する。
    * 管理用ダッシュボード（`canary`配下）に明示的な「データ移行（コピー）ボタン」を設置する。
    * 移行時、v1 の `owned` データを参照し、V2用の最適化されたデータ構造（クリア状況と進行状況）に変換して `users_v2` を初期化する。
* **世代進行型学習**: 算数クイズのクリアにより、1〜9世代を段階的に進行させる。

## 6. 機能要件 (Functional Requirements)
* **IDの二重管理**: マスターデータ内では、各ポケモンに対して「世代内ID（local_id）」と「API ID（api_id）」の両方を保持・管理する。
* **ユーザーデータ構造の最適化**: 過去の取得データをすべて配列で持つのではなく、「クリア済みの世代数」と「現在挑戦中の世代での取得データ」のみを保持する構造とし、データ容量と判定ロジックを軽量化する。
* **コンプリート・世代進行ロジック**: 
    * ユーザーの「現在挑戦中の世代での取得数」と、マスターデータの `pokemon_list.length` が一致した時点でコンプリートとする。
    * コンプリート時、「クリア済みの世代数」を +1 し、「現在挑戦中のデータ」をリセット（空配列に）して次世代へ進行する。
* **図鑑表示・プレビューロジック**: 
    * **開発ダッシュボード内エラーの検知**: 現在 `index_dev_masterdata_2.html` に組み込まれているロード済み図鑑の簡易プレビュー機能において、非同期処理の競合、またはスキーマ解釈の不整合を原因とするエラー（`Cannot read properties of undefined (reading 'name')`）が発生しており、正常に稼働していない。そのため、**ダッシュボード内図鑑機能は別途不具合修正・開発が必要**である。
    * **本番用表示ロジック（予定）**: 「クリア済みの世代」は全ポケモン所持状態として一括表示する。「現在挑戦中の世代」は、個別の取得データに基づき表示/非表示を切り替える。未到達の世代のタブは生成せず、UI上に存在させない。
    * **UIのV1.5踏襲**: 未取得のポケモンは「？」で隠し、取得済みのポケモンはクリック時にモーダルで詳細情報（高さ、重さ、タイプ、フレーバーテキスト）を表示する形式を本実装で再構築する。

## 7. 詳細設計 (Technical Specification)
### 7.1 データ構造 (Firestore)
* **マスターデータ (読切専用)**: `masters/gen_{1..9}` （項目は第4章を参照）
* **V2ユーザーデータ (完全独立)**: `users_v2/{userId}`
    * `cleared_generations` (Number): クリア済みの世代数（初期値: 0）。
    * `current_gen_owned` (Array): 現在挑戦中の世代で取得したポケモンのID（`api_id`）リスト。

### 7.2 ロジックフロー
1. **データ移行（コピー）フロー**:
   `canary` ダッシュボードの「データ移行ボタン」が押された際、以下のロジックで v1 の `users/{userId}/owned` を V2 用に変換し、`users_v2/{userId}` に新規書き込みを行う。
   * **ケースA：v1で151匹フルコンプリートしている場合**
     * 条件: `owned.length >= 151` 
     * 変換結果: `{ cleared_generations: 1, current_gen_owned: [] }`
     * 意味: 第1世代（カントー）はすべてクリア済み扱いとし、アプリ起動時は自動的に**第2世代（ジョウト地方）からスタート**させる。
   * **ケースB：v1が途中の場合（例: 50匹取得）**
     * 条件: `owned.length < 151`
     * 変換結果: `{ cleared_generations: 0, current_gen_owned: [v1のowned配列をそのままコピー] }`
     * 意味: 第1世代のクリア数が0、現在挑戦中のリストにこれまでの取得分を引き継ぎ、**第1世代の残りのコンプリートを目指す状態**からスタートさせる。
2. **クイズフロー**: `cleared_generations + 1` の世代マスターを取得し、`current_gen_owned` に含まれないポケモンを抽出して出題候補とする。
3. **進行判定フロー**: クイズ後、`current_gen_owned.length` と当該世代マスターの `pokemon_list.length` を比較。一致すれば `cleared_generations++` を行い、`current_gen_owned = []` で初期化する。
