# ポケモンさんすう V2 システム定義書（完全・統合版）
*作成日時: 2026年6月1日 08:53 JST*

## 1. プロジェクト目的
* 算数クイズを解くことで1〜9世代のポケモンを順次ゲットし、全世代コンプリートを目指す「世代進行型学習アプリ」へ刷新する。
* 学習進行度に応じてポケモンを段階的に解放し、長期的な学習意欲を向上させる。

## 2. 現状システム (V1.5) の機能要件
* **ユーザー管理**: 事前定義されたユーザー（りょうま、さら、ゲスト）を選択してプレイする形式[cite: 2]。
* **難易度設定**: レベル1（1桁の足し引き）からレベル4（九九）までの4段階のクイズレベルが存在する[cite: 2]。
* **クイズシステム**: 1回につき5問出題され、正解を含む4択から解答を選ぶ[cite: 2]。
* **報酬ロジック**: 4問正解で1匹、5問全問正解で2匹のポケモンを獲得できる[cite: 2]。
* **出現制限**: 選択した難易度によって出現するポケモンの範囲が制限される（レベル1・2は100番まで、レベル3は143番まで、レベル4は151番まで）[cite: 2]。
* **図鑑機能**: 初代151匹固定の図鑑を表示し、未取得のポケモンは「？」で隠される[cite: 2]。取得済みポケモンはモーダルで詳細（高さ、重さ、タイプ、説明文）を閲覧できる[cite: 2]。

## 3. 現状システム (V1.5) の構成と課題
* **データ取得の負荷**: 起動時にPokeAPIから初代151匹分のデータを直接クライアント側で全件フェッチしているため、通信負荷と待機時間が発生している[cite: 2]。
* **拡張性のないデータ構造**: Firestoreの `users` コレクション内のドキュメントに、取得したポケモンのIDを `owned` という単一の配列としてフラットに保存している[cite: 2]。
* **ハードコードされた判定**: コンプリート判定が「取得した一意のIDの数が151に達したか」という固定値で組まれており、世代ごとの動的な変化に対応できない[cite: 2]。
* **次世代機能の未実装**: 151匹コンプリート時に「つぎの せだいへ！」ボタンが表示されるが、押下時の処理はログ出力のみで実体がない[cite: 2]。

## 4. マスターデータ構築（移行準備）の目的とデータ構造
* **目的**: クライアント起動時のPokeAPI直接呼び出しを廃止し、事前にAPIから取得・成形したデータをFirestoreにキャッシュすることで、アプリの起動速度と安定性を向上させる。ユーザーは自身の `current_gen` に対応する世代データのみを読み込む。
* **データ構造**:
    * **ドキュメントパス**: `masters/gen_{1..9}`
    * **フィールド**: `pokemon_list` (Array)
    * **配列内のオブジェクト構造（構築済）**:
        * `local_id` (Number): 世代内の連番ID（1, 2, 3...）
        * `api_id` (Number): PokeAPIの全国図鑑No (`id`)[cite: 2]
        * `name` (String): ポケモンの名前（日本語）[cite: 2]
        * `image` (String): 公式アートワーク等の画像URL[cite: 2]
        * `height` (Number): 高さ[cite: 2]
        * `weight` (Number): 重さ[cite: 2]
        * `types` (Array): 日本語変換済みのタイプ一覧[cite: 2]
        * `flavor` (String): 日本語の説明文（フレーバーテキスト）[cite: 2]

## 5. 要求仕様 (Requirements)
* **独立性の原則（完全不可侵）**: 
    * 既存の v1 環境（`index_v1.5` および Firestore `users` コレクション）に対し、削除・更新・変更等の干渉を一切行わない。
    * v1 データは移行時の参照（`get`）以外、一切のアクセスを禁止する。
* **システム分離**: V2 の運用データは `users_v2` コレクションで完全に独立管理する。
* **データ移行プロトコル**: 
    * ログイン時の自動データ同期は禁止する。
    * 管理用ダッシュボード（`canary`配下）に明示的な「データ移行（コピー）ボタン」を設置する。
    * 移行時、v1 の `owned` データを参照し、V2用の最適化されたデータ構造（クリア状況と進行状況）に変換して `users_v2` を初期化する。
* **世代進行型学習**: 算数クイズのクリアにより、1〜9世代を段階的に進行させる。

## 6. 機能要件 (Functional Requirements)
* **IDの二重管理**: マスターデータ内では、各ポケモンに対して「世代内ID（local_id）」と「API ID（api_id）」の両方を保持・管理する。
* **ユーザーデータ構造の最適化**: 過去の取得データをすべて配列で持つのではなく、「クリア済みの世代数」と「現在挑戦中の世代での取得データ」のみを保持する構造とし、データ容量と判定ロジックを軽量化する。
* **コンプリート・世代進行ロジック**: 
    * ユーザーの「現在挑戦中の世代での取得数」と、マスターデータの `pokemon_list.length` が一致した時点でコンプリートとする。
    * コンプリート時、「クリア済みの世代数」を +1 し、「現在挑戦中のデータ」をリセット（空配列に）して次世代へ進行する。
* **図鑑表示ロジック**: 
    * 「クリア済みの世代」は全ポケモン所持状態として一括表示する。
    * 「現在挑戦中の世代」は、個別の取得データに基づき表示/非表示を切り替える。
    * 未到達の世代（未来の世代）のタブは生成せず、UI上に存在させない。
    * **UIのV1.5踏襲**: 未取得のポケモンは「？」で隠し[cite: 2]、取得済みのポケモンはクリック時にモーダルで詳細情報（高さ、重さ、タイプ、フレーバーテキスト）を表示する形式[cite: 2]を本実装で再構築する。

## 7. 詳細設計 (Technical Specification)
### 7.1 データ構造 (Firestore)
* **マスターデータ (読取専用)**: `masters/gen_{1..9}` （項目は第4章を参照）
* **V2ユーザーデータ (完全独立)**: `users_v2/{userId}`
    * `cleared_generations` (Number): クリア済みの世代数（初期値: 0。v1から完全移行完了時は 1 になる想定）。
    * `current_gen_owned` (Array): 現在挑戦中の世代で取得したポケモンのID（`api_id` または `local_id`）リスト。

### 7.2 ロジックフロー
1. **移行フロー**: `canary` で移行ボタン押下 → v1から取得 → 151匹フルコンプ済みなら `{ cleared_generations: 1, current_gen_owned: [] }` 等、状況に応じた状態に変換して新規作成。
2. **クイズフロー**: `cleared_generations + 1` の世代マスターを取得し、`current_gen_owned` に含まれないポケモンを抽出して出題候補とする。
3. **進行判定フロー**: クイズ後、`current_gen_owned.length` と当該世代マスターの `pokemon_list.length` を比較。一致すれば `cleared_generations++` を行い、`current_gen_owned = []` で初期化する。
