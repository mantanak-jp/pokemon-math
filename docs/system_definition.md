# ポケモンさんすう V2 システム定義書（完全・統合版）

## 1. プロジェクト目的
* 算数クイズを解くことで1〜9世代のポケモンを順次ゲットし、全世代コンプリートを目指す「世代進行型学習アプリ」へ刷新する。
* 学習進行度に応じてポケモンを段階的に解放し、長期的な学習意欲を向上させる。

## 2. 要求仕様 (Requirements)
* **独立性の原則（完全不可侵）**: 
    * 既存の v1 環境（`index_v1.5` および Firestore `users` コレクション）に対し、削除・更新・変更等の干渉を一切行わない。
    * v1 データは移行時の参照（`get`）以外、一切のアクセスを禁止する。
* **システム分離**: V2 の運用データは `users_v2` コレクションで完全に独立管理する。
* **データ移行プロトコル**: 
    * ログイン時の自動データ同期は禁止する。
    * 管理用ダッシュボード（`canary`配下）に明示的な「データ移行（コピー）ボタン」を設置する。
    * 手動操作により v1 の `owned` データを `get` し、V2 用の `users_v2/{userId}` を `{ owned_v2: [取得配列], current_gen: 1 }` として新規作成（初期化）する。
* **世代進行型学習**: 算数クイズのクリアにより、1〜9世代を段階的に進行させる。

## 3. 機能要件 (Functional Requirements)
* **世代進行管理**: 
    * `current_gen` を用いて現在の到達世代を保持する。
    * 該当世代の全ポケモン所持判定時に、システムが自動で次世代を解放（`current_gen` をインクリメント）する。
* **コンプリート判定ロジック**: 
    * 各世代の「ポケモン総数」は、Firestore の `masters/gen_{n}` ドキュメント内の `pokemon_list` 配列の長さ（`length`）を正とする。
    * ユーザーの `owned_v2` に含まれる当該世代のポケモンID数と、マスターデータの `length` を比較し、一致した場合にのみコンプリートと判定する。
* **図鑑表示ロジック**: 
    * `current_gen` を基点とし、第1世代から現在到達している世代までのタブのみを動的に生成・表示する。
    * 未到達の世代（未来の世代）のタブは生成せず、UI上に存在させない。

## 4. 詳細設計 (Technical Specification)
### 4.1 データ構造 (Firestore)
* **マスターデータ (読取専用)**: `masters/gen_{1..9}`
* **V2ユーザーデータ (完全独立)**: `users_v2/{userId}`
    * `owned_v2` (Array): V2で取得した全ポケモンのIDリスト。
    * `current_gen` (Number): 現在の到達世代（初期値: 1）。

### 4.2 ロジックフロー
1. **移行フロー**: 管理者が `canary` ダッシュボードで操作 → `users` から `get` → `users_v2` に `set` で新規データを作成。
2. **クイズフロー**: `current_gen` に紐づく `masters` を取得し、`owned_v2` に含まれないポケモンを抽出して出題候補とする。
3. **進行判定フロー**: クイズ後、`masters/gen_{current_gen}.pokemon_list.length` と、`owned_v2` の当該世代所持数を比較。一致すれば `current_gen++` を実行。
