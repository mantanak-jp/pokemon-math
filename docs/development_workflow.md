# 開発ワークフロー

最終更新: 2026-06-06

## 1. 目的

この文書は、`pokemon-math` リポジトリにおける開発の進め方を定義します。

本プロジェクトでは、iPhone を中心に、GitHub、ChatGPT、Codex Web版を組み合わせて開発を進めます。

現在の本番アプリである root `index.html` は V2 Main 本番版です。そのため、開発では以下を重視します。

```text
- 既存アプリを壊さないこと
- 既存ユーザーデータを壊さないこと
- 本番影響を事前に明示すること
- GitHub上の状態変更操作は、事前承認後にのみ実行すること
- 仕様・設計・実装・レビュー・マージ判断を分離して進めること
```

## 2. 開発環境の前提

基本方針は以下です。

```text
- iPhoneで基本的な開発確認・指示出しを行う。
- GitHub、ChatGPT、Codex Web版を活用する。
- ローカルPC前提の開発フローにはしない。
- npm / build / local server が必須となる構成は、当面採用しない。
- GitHub Pagesでそのまま動く静的HTML / CSS / JavaScript構成を維持する。
```

将来的にデスクトップ版Codexやリモート接続型の開発体験を使える場合は、別途検証します。

ただし、現時点では以下を基本フローとします。

```text
ChatGPT:
要求仕様・機能要件・詳細設計・PR作成・差分レビュー支援

Codex Web版:
コード生成・コード修正案作成・リファクタリング案作成

GitHub:
リポジトリ正本管理・ブランチ管理・PR管理・マージ管理

ユーザー:
仕様判断・状態変更操作の承認・実機確認・最終マージ承認
```

## 3. 役割分担

### 3.1 ChatGPTの役割

ChatGPTは主に以下を担当します。

```text
- 要求仕様の整理
- 機能要件の整理
- 詳細設計の作成
- 影響範囲の整理
- Codex向け実装指示の作成
- GitHub上の既存ファイル・差分確認
- PR作成
- PR本文作成
- レビュー観点の整理
- マージ可否判断の支援
- 次チャットへの引き継ぎ文書作成
```

ChatGPTは、読み取り以外のGitHub操作をユーザー承認なしに実行してはいけません。

### 3.2 Codex Web版の役割

Codex Web版は主に以下を担当します。

```text
- ChatGPTで整理した仕様・設計に基づくコード生成
- 既存コードの修正案作成
- HTML / CSS / JavaScript の分割・整理
- コードレビュー用の差分説明
- 必要に応じたテスト観点の提示
```

Codex Web版からGitHubへのPR作成ができない場合、PR作成はChatGPTが担当します。

### 3.3 GitHubの役割

GitHubは以下を担います。

```text
- mainブランチの正本管理
- 作業ブランチ管理
- Pull Request管理
- Files changed による差分確認
- ユーザーによる最終マージ承認
- merged / closed PR の履歴管理
```

### 3.4 ユーザーの役割

ユーザーは以下を担当します。

```text
- 仕様の最終判断
- GitHub上の状態変更操作の承認
- iPhone実機での動作確認
- 実データに関わる判断
- GitHub上での最終マージ承認
```

## 4. 基本開発フロー

標準的な開発フローは以下です。

```text
1. ユーザーが要望・課題を提示する
2. ChatGPTが要求仕様・機能要件・詳細設計を整理する
3. ユーザーが内容を確認する
4. 必要に応じてChatGPTがCodex向け指示文を作成する
5. ユーザーがCodex Web版へ指示文をコピー＆ペーストする
6. Codexがコード修正案を生成する
7. ユーザーがCodex出力をChatGPTへ共有する
8. ChatGPTが差分・影響範囲・禁止事項との整合を確認する
9. ChatGPTがPR作成前説明を行う
10. ユーザーがPR作成を承認する
11. ChatGPTがGitHub上でPRを作成する
12. ChatGPTがPR内容を再確認する
13. ユーザーがGitHub上で確認・マージ承認する
14. 必要に応じてChatGPTが次フェーズの引き継ぎを整理する
```

## 5. 状態変更操作の扱い

以下はすべて状態変更操作として扱います。

```text
- ブランチ作成
- ファイル作成
- ファイル更新
- ファイル削除
- PR作成
- PR更新
- PRクローズ
- ブランチ削除
- mainへのマージ
- Firestoreデータの作成・更新・削除
```

ChatGPTは、状態変更操作の前に必ず以下を説明します。

```text
- 実行対象
- 実行内容
- 影響範囲
- 後戻り可否
- 変更されないもの
```

そのうえで、ユーザーに明示的に確認します。

```text
実行してよいですか？
```

ユーザーが明示的にOKした場合のみ実行します。

## 6. 禁止事項

以下は禁止します。

```text
- 承認前にGitHubのファイルを作成・更新・削除する
- 承認前にPRを作成する
- 承認前にブランチを削除する
- 承認前にmainへマージする
- root index.html を不用意に変更する
- users コレクションへV2 Main通常プレイ用の書き込みを追加する
- masters/gen_{1..9} へ通常プレイ中の書き込みを追加する
- canary/migration.html への通常ユーザー導線を追加する
- 仕様未確定のまま本番コードへ変更を入れる
```

## 7. PR作成前チェック

PR作成前には、以下を確認します。

```text
- 変更対象ファイルは想定どおりか
- root index.html の変更有無
- Firestore読み書き先の変更有無
- users / users_v2 / masters の扱いが仕様どおりか
- migration.html への通常導線が追加されていないか
- UI文言が子ども向け仕様と矛盾していないか
- 図鑑モーダル仕様が戻っていないか
- body.modal-open を復活させていないか
- 保存失敗時の「もういちどゲットする」が維持されているか
- 通常結果画面の「同じレベルでつづける」「ずかんをみる」「メニューへもどる」が維持されているか
```

## 8. PR本文に明記する事項

PR本文には、最低限以下を記載します。

```text
## Summary
- 何を変更したか

## Scope
- 変更対象ファイル
- 本番影響の有無
- Firestore影響の有無

## Checks
- 差分確認結果
- index.html変更有無
- users / users_v2 / masters の扱い
- migration.html通常導線なし
- 実機確認予定または実施結果
```

`index.html` を変更する場合は、特に以下を明記します。

```text
- 本番V2 Mainへの影響
- 変更した画面・機能
- 既存ユーザー ryoma / sara に影響しないこと
- guestで確認すべき項目
```

## 9. 実機確認方針

実機確認は、原則としてiPhoneで行います。

優先順位は以下です。

```text
1. guest
   - 通常プレイ確認
   - 保存確認
   - 図鑑確認
   - 世代クリア検証が必要な場合の検証用

2. ryoma / sara
   - 原則として不用意に進行データを変更しない
   - 表示確認のみを基本とする
```

実データ保護のため、`ryoma` / `sara` でのクイズ実行・ポケモン取得・保存確認は慎重に判断します。

## 10. Codex連携ルール

Codex Web版に渡す指示は、以下を含めます。

```text
- 作業目的
- 対象ファイル
- 変更してよい範囲
- 変更してはいけない範囲
- 既存仕様
- 禁止事項
- 出力形式
- 差分説明の依頼
```

Codexへの指示では、以下を明示します。

```text
- mainへ直接反映しない
- 仕様外の改善を勝手に追加しない
- Firestoreデータ構造を勝手に変えない
- users / users_v2 / masters の扱いを変えない
- migration.htmlへの通常導線を追加しない
```

## 11. ドキュメント更新ルール

以下の場合は、コード変更と同時に関連ドキュメントを更新します。

```text
- Firestoreの読み書き先を変更する場合
- V1.5 / V2 のデータ構造を変更する場合
- 画面遷移を変更する場合
- UI文言を変更する場合
- 報酬仕様を変更する場合
- クイズ仕様を変更する場合
- index.html / canary / archive の位置づけを変更する場合
- リリース運用を変更する場合
```

関連ドキュメントの例:

```text
- docs/current_inventory.md
- docs/data_model_v1_v2.md
- docs/migration_v1_to_v2.md
- docs/system_definition_v2.md
- docs/ui_ux_v2.md
- docs/release_and_canary.md
- docs/v2_canary_handoff.md
```

## 12. 今後の改善候補

今後、開発フロー自体を改善する候補は以下です。

```text
- Codexのリモート操作機能の検証
- iPhoneからCodex作業を監視・承認する運用の検証
- コピペ連携を減らす方法の検討
- PRテンプレートの追加
- docs/review_checklist.md の追加

```

ただし、現時点では、安定した手動承認フローを優先します。
